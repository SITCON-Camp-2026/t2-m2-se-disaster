# 需求取捨決策

> 這份文件是 Codex 先產生的草稿，最後必須由學員刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [x] 回報者
- [ ] 資訊整理者
- [ ] 行動者

原因：我想優先服務回報者，因為資訊一開始進來時如果沒有保留「不確定」「只是轉述」「不知道」這些狀態，後面整理者很容易把模糊線索誤當成已確認事實。v1 先讓回報者能快速留下資訊，同時清楚標示哪些內容還需要人工確認。

## 這個使用者最想完成什麼

- 快速把現場看到或聽到的狀況留下來。
- 清楚標示自己是親眼確認，還是只是轉述別人的資訊。
- 在不確定時保留「不知道」「可能」「聽說」這些描述，不要硬填成確定答案。

## 這個使用者最怕什麼錯誤

- 把聽說或轉述的資訊寫得像自己親眼確認。
- 為了讓回報看起來完整，硬填不確定的地點、數量或需求類型。
- 回報內容被後續流程誤解成已確認資料或正式任務。

## 我今天暫時不處理的需求

- 暫時不做真正的新回報送出後端，只先設計前端流程和狀態。
- 暫時不做行動者派工、出發、路線、地圖或現場安全判斷。
- 暫時不處理帳號登入、回報進度追蹤、照片上傳、地圖定位或自動通知。

## 我不採用的 AI 建議

- AI 建議：讓系統根據高信心候選結果判斷是否可以採取行動。
  - 不採用原因：這會把 AI 推論變成救災決策，不符合課程安全邊界。
- AI 建議：把相對完整的資訊顯示成可用或已確認。
  - 不採用原因：只要原始資料仍是 `needs_review` 或 `unverified`，就不能顯示成已確認。

## 必須保留人工確認的判斷

- 轉述者是否親眼確認。
- 模糊地點是否足以讓人定位。
- 資訊是否過期或和其他回報衝突。
- 涉及個人位置、健康、長者或弱勢個案的資料是否可以公開。
- 某筆資訊能不能變成志工任務。

## 今天明確不做什麼

- 不做真實救災派工。
- 不補真實地址、電話、人物或地圖資料。
- 不把 Phase 0 原始資訊整理成正式資料。
- 不修改 `src/`，Release 01 只做需求分析文件。

## 我仍不確定的地方

- 我還不確定要怎麼平衡「快速回報」和「資料足夠清楚」。
- 我還不確定 v1 要先改善回報流程，還是先改善整理畫面。
- 我還不確定哪些提醒文字最能避免回報被誤解成已確認任務。

## Release 02 流程設計後的補充

- 流程草稿改以回報者為入口，因為目前討論希望使用者一進 v1 先看到可以填寫資訊的畫面。
- 這個方向和 Release 01 原本暫勾的「資訊整理者」不同；目前先採用回報者，但仍可在小隊討論後調整。
- 即使流程從回報者開始，送出內容仍只能是原始資訊，必須進入等待人工審核，不能直接變成已確認資料或志工任務。
