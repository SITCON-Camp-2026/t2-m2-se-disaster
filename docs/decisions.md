# 需求取捨決策

> 這份文件是 Codex 先產生的草稿，最後必須由學員刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：目前 Phase 0 已經有一批原始資訊與整理草稿，最大的風險是下一位協作者把候選判斷、示範草稿或 AI 推論看成已確認事實。v1 暫時優先服務資訊整理者，讓「不能判斷的理由」和「需要人工確認」更清楚。這是 AI 草稿，請學員確認或改選。

## 這個使用者最想完成什麼

- 快速分辨哪些原始資訊只是線索，哪些已有整理草稿。
- 保留原文、整理判斷、推論和阻礙理由之間的界線。
- 讓下一位協作者知道為什麼某些資訊不能直接變成任務。

## 這個使用者最怕什麼錯誤

- 草稿或候選判斷被看成已確認資料。
- AI 補了原文沒有的地點、需求、數量或行動建議。
- 查核狀態、任務狀態、來源類型混在一起。

## 我今天暫時不處理的需求

- 暫時不做真正的回報表單，也不處理新資料送出流程。
- 暫時不做行動者派工、出發、路線、地圖或現場安全判斷。
- 暫時不新增後端、資料庫、外部 API 或 runtime LLM。

## 我不採用的 AI 建議

- AI 建議：讓系統根據高信心候選結果判斷是否可以採取行動。
  - 不採用原因：這會把 AI 推論變成救災決策，不符合課程安全邊界。
- AI 建議：把相對完整的資訊顯示成可用或已確認。
  - 不採用原因：只要原始資料仍是 `needs_review` 或 `unverified`，就不能顯示成已確認。

## 必須保留人工確認的判斷

- 轉述者是否親眼確認。
- 模糊地點是否足以讓人定位。
- 資訊是否過期或和其他回報衝突。
- 涉及個人位置、健康、長者或弱勢個案的資料是否可以公開。
- 某筆資訊能不能變成志工任務。

## 今天明確不做什麼

- 不做真實救災派工。
- 不補真實地址、電話、人物或地圖資料。
- 不把 Phase 0 原始資訊整理成正式資料。
- 不修改 `src/`，Release 01 只做需求分析文件。

## 我仍不確定的地方

- v1 應該優先改文案、資訊架構，還是草稿編輯流程。
- 「資訊整理者」是否真的是小隊共識，待小隊討論後確認。
- 哪些標籤最容易讓使用者誤會成已確認，待實際操作觀察。
