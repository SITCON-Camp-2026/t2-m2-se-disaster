# Phase 0 Observations

## 我們做出了什麼？

- 我們做出一個 Phase 0 前端整理工作台，可以從首頁切到「原始資訊」與「整理工作台」。
- 工作台會顯示每筆原始資訊的資訊取得方式、查核狀態、示範草稿 / 學員草稿來源，以及不能直接行動的原因。
- 目前草稿仍是學習用候選判斷，不是整理後資料，也不是已確認結果。

## 哪些資料讓 agent 很容易整理？

- 有明確時間、地點描述、數量或限制條件的資訊比較容易整理，例如物資數量、服務狀態或公告同步問題。
- 來源是現場回報或志工更新時，agent 比較容易整理成「候選地點狀態」或「候選更新」。
- 即使看起來容易整理，也只能說「原文提供較多線索」，不能直接說已確認。

## 哪些資料讓 agent 亂補或誤判？

- 位置只寫「老雜貨店後面」「第二排住家」這類模糊描述時，agent 可能會自行補成看似可派工的地點。
- 原文只說「需要水電」時，agent 可能會推測成檢修、供電或修復，但原文沒有說清楚是哪一種。
- 原文提到「先不要再派人」但沒有說原因時，agent 可能會猜是人力過多、道路危險或任務完成；這些都需要人工確認。
- 涉及長者、患者或住家位置時，agent 可能只看到任務需求，忽略當事人同意與隱私風險。

## 哪些資訊不能直接變成任務？

- 位置不明確、找不到當事人或不知道現場是否仍需要協助的資訊，不能直接變成志工任務。
- 只有需求類型但沒有工作範圍、時間、工具需求或安全條件的資訊，不能直接派人。
- 「不要派人」這類阻擋訊息不能被反向解讀成需求已消失；必須先確認原因。

## 哪些資訊的操作者不是當事人？

- 志工代為轉述的住家或長者需求，需要確認當事人是否同意公開資訊與接受協助。
- 社群轉錄或電話口述可能不是第一手資訊，不能把轉述者當成當事人。
- 由現場人員貼出的公告或庫存狀態，需要分清楚「資訊整理者」與「決策者」不是同一個人。

## 我們剛剛偷偷假設了什麼？

- 我們假設示範草稿中的整理方向是合理的，但它仍需要人類檢查，不是標準答案。
- 我們假設有具體數量或時間的資訊比較可靠，但來源類型和查核狀態仍要分開看。
- 我們假設「不能直接行動」可以靠 UI 標籤降低誤用風險，但真正的風險仍需要流程與人工確認補上。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要更明確的人工確認流程，定義誰可以確認、確認什麼、確認後如何留下紀錄。
- 需要讓每個候選判斷保留原文依據與人類修正理由，避免只留下整理後文字。
- 需要能標出「暫時不採用」與「需要補問」的原因，讓下一位協作者知道不是漏處理。

## 我們需要更清楚資料格式的地方

- 需要分開記錄資訊取得方式、查核狀態、草稿來源、人工確認狀態，不能混在同一個欄位。
- 需要記錄阻礙因素，例如地點不足、當事人未同意、需求範圍不明、時間可能過期。
- 需要保留原始文字，讓人能回頭檢查 agent 是否補了原文沒有的內容。

## 我們需要人工確認的地方

- 確認模糊地點是否足以安全找到現場。
- 確認當事人是否同意公開個人位置或接受協助。
- 確認需求是否仍存在、是否已被處理、是否有安全限制。
- 確認 agent 草稿中的候選類型、下一步建議與阻礙因素是否只是推測。
